# Orval Setup â€” React Query + Zod Code Generation

## Overview

Set up Orval to auto-generate React Query hooks, API functions, and Zod validation schemas from the backend OpenAPI spec. This replaces all hand-written API functions, React Query hooks, and TypeScript interfaces with generated code.

## Input

- **OpenAPI spec URL:** `http://100.81.155.97:8000/api/schema/`
- Fetched live on each generation run

## Orval Configuration

Single `orval.config.ts` at project root with one named target (`compliance`):

```ts
import { defineConfig } from 'orval'

export default defineConfig({
  compliance: {
    input: {
      target: 'http://100.81.155.97:8000/api/schema/',
    },
    output: {
      target: 'src/api/generated/',
      client: 'react-query',
      mode: 'tags-split',
      schemas: 'src/models/',
      prettier: true,
      override: {
        mutator: {
          path: 'src/api/mutator/custom-instance.ts',
          name: 'customInstance',
        },
        zod: {
          generate: {
            response: true,
            body: true,
            param: true,
            query: true,
            header: false,
          },
        },
        header: (info) => [
          `Generated by orval`,
          `Do not edit manually.`,
          ...(info.title ? [info.title] : []),
        ],
      },
    },
    hooks: {
      afterAllFilesWrite: 'prettier --write',
    },
  },
})
```

### Key Configuration Notes

- **`client: 'react-query'`** â€” generates `useQuery` / `useMutation` hooks per endpoint
- **`mode: 'tags-split'`** â€” one directory per API tag. If backend endpoints are untagged, they go to a `default/` directory
- **`schemas: 'src/models/'`** â€” Zod schemas + inferred TS types generated here
- **`override.zod.generate`** â€” enables Zod validation for response bodies, request bodies, path params, and query params
- **`override.mutator`** â€” all generated hooks delegate HTTP calls to the custom instance (which wraps the existing `apiClient`)

## Custom Mutator

File: `src/api/mutator/custom-instance.ts`

```ts
import apiClient from '@/api/api-client'
import type { AxiosError, AxiosRequestConfig } from 'axios'

export const customInstance = <T>(config: AxiosRequestConfig): Promise<T> => {
  const promise = apiClient(config).then(({ data }) => data)
  return promise
}

export default customInstance

export type ErrorType<Error> = AxiosError<Error>
```

- Wraps the existing `apiClient` (auth tokens, base URL, interceptors all preserved)
- Returns unwrapped `response.data`
- Exports `ErrorType` for typed error handling in generated hooks

## Next.js App Router: `'use client'` Directive

Generated hooks use `useQuery`/`useMutation` which are client-side only. Orval's `override.header` is used to add the `'use client'` directive is NOT needed in the generated files themselves â€” the consuming page/component files already have `'use client'` where needed per the existing project convention.

## ESLint Configuration

Add `src/api/generated/` and `src/models/` to ESLint's `globalIgnores` in `eslint.config.mjs` to prevent linting generated code.

## Migration Strategy

All-at-once replacement:

1. Install Orval and run generation
2. Create the custom mutator
3. Update all import paths in consuming components (pages, column definitions, etc.) to use generated hooks and types
4. Remove the old hand-written files listed above
5. Verify with `yarn type-check` and `yarn build`

### Parameter naming

Generated hooks will use parameter names matching the OpenAPI spec (likely `snake_case` e.g. `page_size`). Existing code already passes `page_size` to the API functions, so no parameter renaming is needed.

### Mutation side-effects

Existing mutation hooks (e.g. `useCreateTicket`) use `onSuccess` callbacks for cache invalidation via `queryClient.invalidateQueries()`. After migration, consumers will pass `onSuccess` callbacks directly to the generated mutation hooks' options parameter, which React Query supports natively.
